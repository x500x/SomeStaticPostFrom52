说到Windows上的文件预分配，似乎是个比较小众的话题，一些有经验的用户可能会有点印象，比如BT软件的高级设置中可能会有（比如qBittorrent“设置”--“下载”选项卡里的“为所有文件预分配磁盘空间”），还有FastCopy的介绍里记得有“特权模式下预分配空间”什么的，知道fsutil file createNew命令的可能就更少了。

磁盘空间的预分配简单来说，就是在磁盘空间里“快速圈地”，直接操作磁盘空间的分配表，提前一次性划出一块尽可能连续的区域供程序读写，这样只要在这个区域里进行读写，就不用牵扯到申请额外的空间。不像从空白文件开始写入那样，每次申请额外增长的空间时，都有可能分到随机地址的空间（想要相邻空间时，相邻空间已经被其他文件占用或预定；或者受其他的操作系统空间分配策略的影响），有一定概率分到非相邻的空间，从而造成磁盘碎片。

经过我这差不多一个星期的研究和测试，算是搞出来了一个专门用于在windows上进行文件预分配的python函数。
先不多说废话，直接上最终的代码和使用说明，然后再来讲一下故事。

（4）solid_mode(int) ：“虚/实模式”选择
默认值：-1
------------------------------------------------------------
-1：自动根据文件系统的类型选择
（NTFS或ReFS会选择“实模式”，其他FAT系列的文件系统会选择“虚模式”）
 0：关闭“实模式”，使用“虚模式”
 1：启用“实模式”
------------------------------------------------------------
# “虚模式”： 
#     仅指定文件的磁盘空间分配大小（FileAllocationInfo）
#     关掉文件句柄后，文件的最终大小由写入时“够到”的最远写入距离决定，
#     而不是文件一开始的分配大小。
#   优点：不受文件系统影响，在各个windows原生支持读写的文件系统上均支持，
#         相比于“实模式”在一开始文件就占据确定的空间，
#         虽然写入完成前EOF的位置一直在变，看上去好像状态比“实模式”不确定些，
#         但系统确实已经为文件提前划定好了空间，只要是在文件句柄存续期间（单次）写入，
#         写出的文件产生碎片的概率极小。【适合单次连续写入】
#   缺点：不过如果是关掉文件句柄再重新打开进行追加写入（比如日志），
#         即便和第一次一样分配相同的空间（两次分配的空间均大于两次文件的实际大小），
#         第二次也有可能会分到和第一次不相邻的区域，从而产生文件碎片。
#          
# “实模式”：
#     在“虚模式”指定文件的磁盘空间分配大小（FileAllocationInfo）的基础上，
#     进一步修改FileEndOfFileInfo，移动EOF至分配的大小，使文件在一开始就占据确定的空间。
#     关掉文件句柄后，只要不在最后进行“截断”操作，文件的最终大小就是文件一开始的分配大小。
#   优点：文件在一开始就占据确定的空间，后续重新打开文件追加写入时只要在第一次预分配的范围内，
#         就不会产生像“虚模式”那样可能分到非邻近区域的问题。
#   缺点：由于windows原生支持记录“数据有效长度（VDL）”的文件系统测试下来仅有NTFS和ReFS，
#         所以如果在较老的FAT文件系统上采用“实模式”，
#         文件句柄关闭时系统将强制给没写入的地方“填零”，如果预分配的空间被写入的部分不占绝大部分的话，
#         整个进程极有可能会因为“填零”卡死在那很长一段时间。
#
#   ★ 以上优缺点都是基于连续不间断写入的前提下，如果是多线程、随机、跳转的写入，不用管理员权限解开“数据有效长度（VDL）”限制的话，
#      我这边测试过，文件句柄每次向后seek到“数据有效长度（VDL）”外的下一写入点时，系统都会先将前面未写入的部分强制填零，
#      然后才会seek到下一个写入点进行写入，几乎每次都要卡一段时间，除了极少被使用的ReFS能避免（估计是因为ReFS支持追踪多个“数据有效区域”）
#      和“实模式”还是“虚模式”无关。
#


（5）vdl_mode(int) ：是否获取文件全部长度的“数据有效长度（VDL）”
默认值：-1
------------------------------------------------------------
-1：自动选择
（一般默认关闭，除非在FAT系列文件系统上，
且“solid_mode==1”的情况下，会尝试启用，如果没有管理员权限会抛出错误）

# 为多线程、随机、跳转写入解除强制填零的障碍，但预分配范围内可能会引入“脏数据”（以前删除的文件数据），

# NTFS、ReFS则不进行rewind，仅关闭句柄后尝试删除文件。

我看里面还提到：“The preallocated disk space is also released when you call SetEndOfFile”（调用 SetEndOfFile 时，也会释放预分配的磁盘空间。）

思考了下，如果把这两步结合到一起，先分配磁盘空间，然后移动EOF到指定的分配大小，说不定就能创建好一个空白的用来占位的文件。于是说干就干，手搓出了第一版脚本。简单测试了下，不仅速度嗷嗷快，而且完全不需要管理员权限。在win7虚拟机上测试了，32、64位的都试过了，即便UAC设置拉到顶，脚本运行中也不会触发烦人的UAC弹窗。兴高采烈正准备写文章发帖，结果猛然想到目前只测试了NTFS这一种文件系统，不测试一下其他文件系统怎么行。

于是我插上tf卡和转接器准备测试一下exFAT文件系统，然后就出乎意料的卡住了，我以为是转接器的问题，又把家里的闲置的老移动硬盘拿了出来，格式化成exFAT，一试还是这样，连用fsutil创建空白文件也会卡。看来还是方法有点问题。于是我想到NTFS似乎有个叫“空白空间映射”的高级特性（后来深入了解后才知道是“有效数据长度（VDL）”，估计“空白空间映射”完全是另一个概念了），exFAT似乎不支持这个，创建空白文件的时候估计会强制写入“0x00”数据块来填充，测了下时间，刚好和我这块移动硬盘70～80MiB/s的最大读写速度对的上。

为了验证卡住的这段时间是用在填零上的，我在VMware虚拟机里分出来一个4G稍大的分区，先用Diskgenius用随机值填充这个分区的扇区数据，然后用我的脚本在这个分区里创建一个文件出来，然后用diskgenius查看这个文件对应的区域，发现果然是这样，周围都是随机值乱糟糟的，就这个文件里是清一色的“0x00”，实锤了。

这下麻烦了，脚本能不能顺利运行还得看文件系统类型，属实让人头疼。现在也没什么好办法，要不就根据那篇文档试试只动文件的分配空间“FileAllocationInfo”，不移动EOF？于是用了pywin32里win32file.WriteFile试了试，在win32file.CreateFile生成的句柄存续期间用os.urandom生成随机数据写进去，最后用win32file.CloseHandle关掉，果然在exFAT上不卡了。不过不是python自己的文件句柄用起来不太方便（不能套with），于是我之后找到了msvcrt.get_osfhandle这个函数，这样由python的open创建出python自己的文件句柄，然后get_osfhandle根据文件描述符fileno，得到此时这个打开的文件的windows格式的文件句柄，然后win32file.SetFileInformationByHandle就可以操作了。

用这个只动文件的分配空间“FileAllocationInfo”的脚本在各个文件系统上试了试，效果很好，没出现卡顿的同时，输出的文件也是没有碎片的，看来问题就出在移动EOF的时候。

我当时以为EOF在向前移动的同时系统在同步填零，所以后面发现SetFileValidData这个函数，看到设定值要小于等于文件大小（不能超过EOF的位置）时心想这也没用啊，EOF移动的时候就填零了，还轮得到SetFileValidData什么事，不过还是抱着试试看的心态决定试试这个函数。

用SetFileValidData这个函数的时候也遇到了不少坑，一开始我以为自己系统超级管理员“Administrator”的身份调用这个函数绝对够用了，结果一直返回0，后来查了资料，用“WHOAMI /PRIV”命令（其实这也是后面发现的命令，我当时是直接用的win32security.GetTokenInformation，根据返回的LUID和代表启用状态的一长串元组查的）才发现 SE_MANAGE_VOLUME_NAME 特权即使是管理员也是默认关闭的。好不容易摸索出来修改安全令牌token的函数win32security.AdjustTokenPrivileges的用法，结果还放错位置，放在了文件句柄打开后的地方（也是后来才发现的，当时瞎猫碰死耗子调用SetFileValidData成功后一段时间看的资料才知道的）。

回到刚才说要试试SetFileValidData的这个地方，我先直接没有移动EOF就调用SetFileValidData，结果文件大小为零，显然失败了，我略显无奈地只好加上移动EOF的代码，然后再调用它，想着这回得等好一会儿让EOF移动完了，没想到出人意料的是刚一按下回车就搞定了，文件大小完全和预期一样，而且用winhex打开后也能发现里面也有之前删除的文件残留的部分，似乎是成功了。啊这…这也行？在移动EOF的代码后面加了一行os.system("pause")，发现移动EOF很快就完成了，真正卡住的是后面的地方，想想也就只有不起眼的.close()了，让python打印了下耗时，果真大部分时间全耗在了close上面，只要调用SetFileValidData解锁成功，后面的close就不会卡住。

到此豁然开朗，对函数又是一阵修修补补，最终成就了现在放在前面300多行代码的函数。
在此期间还发现，虽然通过移动EOF在NTFS上创建一个实实在在占用空间的空文件很快，但是一旦不是从头到尾连续写入，像是随机、跳转写入、多线程写入，就会因为填零而卡住，原因应该是NTFS仅支持记录固定从文件开头开始的一段“数据有效长度（VDL）”，跳过一段距离写入时，为了留住前面新写入的数据，就需要把“数据有效长度（VDL）”扩展到新写入的数据尾部，由于新扩展的部分中包括了跳过部分的未知空间（存在“脏数据”），只能选择把跳过的部分“推平”（没法新增一段“数据有效长度（VDL）”），这应该不难理解。不过在ReFS上似乎就没有这个问题，测试在这种通过单纯移动EOF来创建的文件上跳转写入时，观察过每一步的耗时，没有发现卡顿迹象，感觉ReFS似乎就支持记录多段“数据有效长度／区域”。这里也不是推荐各位使用ReFS，现在网上对它的评价是故障率高、数据极难恢复、很难用、不兼容、没法缩小分区大小……感觉ReFS就是个实验品。
